## 0. 一句话目标 & 反模式

**目标:让 Lumen 在终端里"可靠地写出好代码 + 用着舒服"——对标 Claude Code / Cursor 的体验,不是行数。**

**反模式(立刻停掉):`ROADMAP-13.md` 那条"31K → 70K 行 / 13 轮加包"的路线。**
- 现在 `internal/` 已经 **192 个包 / 414 个 go 文件**。一个终端编码 agent 不需要 `bloom / graphdb / vectorstore / apigateway / clustermap / circuitbreaker / hermes / maestro` 这种广度。
- 行数和包数是**虚荣指标**。再加只会让"能编译、能跑对、好维护"更难。
- 新北极星指标:**(a) 编辑/写代码的正确率;(b) 端到端可靠性(能编译、测试绿、不崩);(c) 交互体验(流式、diff、输入、状态)。**

## 2. 优先级路线(环环相扣,但按价值排,不按行数)

| 阶段 | 目标 | 为什么先做它 | 主要 owner |

| **P1 可靠性地基** | 修 lsp 类型 bug;消除 modelpool flake;给**核心**包(agent/tool/edit/session/permission/tui)补测试 + `-race`;砍掉明显死包(见 P3 起步)。 | 编码 agent 的信任来自"不崩、改对"。 | Claude 定方向 + DeepSeek 补机械测试 |
| **P2 编码质量闭环** | **edit 正确性**(精确 diff、失败回滚)+ **verify-after-edit**(改完自动 build/test,把结果喂回 loop)+ LSP 真接(诊断/跳转/补全喂给模型)。这是对标 CC/Cursor 的命门。 | 这是"写出好代码"的核心能力差距。 | Claude(架构)+ DeepSeek(语言适配/fixtures) |

## 3. 分工原则(怎么判断给谁)

**给 Claude(难/不可机械化):**
- 触碰 **agent tool-loop / 流式 / 上下文压缩 / checkpoint** 的任何改动。
- **安全**(权限模式、bash guard、文件边界)、**架构决策**、**删哪个包**。
- **有歧义的 debug**(lsp 类型不匹配、flake 根因、编译断裂根因)。
- 写 **DeepSeek 的任务卡**本身 + **review** DeepSeek 产出。

**给 DeepSeek(机械/规格明确/可并行):**
- 按**明确规格**补测试(Claude 给出待测函数 + 期望表)。
- 语法高亮加语言、golden fixtures、表驱动用例。
- 规格清晰的机械重构(重命名、抽函数、按模板补样板)。
- 文档/注释生成(基于已定结构)。

**判据一句话:** 答案唯一、可由验收标准机械判定 → DeepSeek;需要判断/取舍/根因 → Claude。

## 4. 启动任务卡(可直接开干)

> 卡片放 `docs/tasks/`(沿用现有约定)。每张卡必须有:**目标 / 文件 / 验收标准 / owner**。

### 🟥 Claude 起步卡(按顺序)

**C-1 修复编译(P0,最高优先)**
- 文件:`cmd/lumen/main.go`(:219 `model.WaitInput`、:223 `time`),`internal/tui/`。
- 做:查 `74023c0`/`d71cf4b` 重构后 TUI 的真实输入 API,把 `main.go` 接到新方法(可能是 lineedit 的 `ReadLine`);补 `import "time"`。
- 验收:`GOTOOLCHAIN=local GOFLAGS=-mod=mod go build ./...` 退出 0;`go vet ./...` 无新错;`./bin/lumen run "echo hi"` 能起。

**C-2 立"编译+vet"硬门**
- 做:CI(或 pre-commit / Makefile `check`)跑 `go build ./... && go vet ./...`,失败即红。
- 验收:故意写个编译错能被门拦下。

**C-3 修 lsp 真 bug**
- 文件:`internal/lsp/lsp_real.go` + 测试。
- 做:`documentation`/诊断按 `string | object` 两种都解析;并发跑不 `signal: killed`(限并发/超时)。
- 验收:对本机 gopls v0.16.2 跑 lsp 测试全绿,可重复 3 次。

**C-4 消除 modelpool flake**
- 做:定位全量并行下的竞态(`-race`),修共享态。
- 验收:`go test -race -count=5 ./internal/modelpool/` 全绿。

**C-5 P2 设计:verify-after-edit 闭环**
- 做:写设计 spec(`docs/superpowers/specs/`):编辑后自动选 build/test 命令、解析结果、把失败喂回 agent loop 让它自修。定义接口 + 给 DeepSeek 拆机械卡。
- 验收:spec 评审通过 + 至少 3 张可执行 DeepSeek 卡。

**C-6 P3 瘦身盘点**
- 做:对 192 包出一张"留/删/合并"表(标准:是否服务终端写代码 + 是否被 agent 主路径引用)。先删 3-5 个明确死包(无引用/纯 demo)。
- 验收:删后 `go build ./...` 仍绿;包数下降;表归档。

### 🟦 DeepSeek 起步卡(C-1 合并后才发,且彼此并行安全)

**D-1 语法高亮补语言**
- 文件:`internal/render/`(highlighter)。
- 做:按现有表驱动结构,加 Rust / Java / C++ / SQL 的关键字+注释+字符串规则。
- 验收:每语言一组 golden fixture,`go test ./internal/render/` 绿。

**D-2 render 包 golden fixtures**
- 做:为 markdown→ANSI、diff、TruncateVisible 补输入/输出 golden 用例。
- 验收:覆盖 ≥ 主要分支;`go test` 绿。

**D-3 核心包表驱动测试(Claude 指定清单)**
- 范围:Claude 在 C 卡里点名的包(预期 `internal/lineedit`、`internal/render`、`internal/contextmgr` 的纯函数)。
- 做:按 Claude 给的"输入→期望"表写表驱动用例。
- 验收:`go test -race` 绿;不改生产代码(只加测试)。

> ⚠️ DeepSeek 卡的铁律:**不碰 agent loop / 安全 / 架构;只动 Claude 点名的文件;只加测试或按模板补码;产出由 Claude review 后才合。**

## 5. 流程与质量门

- **TDD**:先写失败测试,再实现(尤其 C-3/C-4 的 bug,先复现再修)。
- **每次改完必跑**:`GOTOOLCHAIN=local GOFLAGS=-mod=mod go build ./... && go vet ./... && go test -race ./<改动包>/`。
- **`-race` 是默认**,不是可选(这是个并发 agent)。
- **不新增 internal 包**,除非 Claude 在卡里明确批准(对抗 §4 的膨胀)。
- **DeepSeek 产出 100% 过 Claude review** 才合;DeepSeek 不自行决定接口/删包。
- **小步提交**:一卡一 PR/commit,信息写清"做了什么 + 验收怎么过的"。
- **报告归报告,真相归 `go test`**:不再往 AUDIT/ASSESSMENT 里加夸张结论。
