## 4. Dogfood 方法(验证的锤子 —— ultracode 必用)

> **DeepSeek 余额已于 2026-06-19 充值,可实跑。** `lumen run "<task>"` = 非交互一次性,最适合从 shell 里 dogfood。交互 TUI 需真 tty(非-tty 的 harness 驱动不了 → 渲染函数靠单测验证,整体行为靠 `lumen run` 验证)。

```bash
export PATH="$HOME/.local/bin:$HOME/sdk/go/bin:$HOME/sdk/pg/bin:$HOME/.bun/bin:$PATH"
# 1) 干净 worktree(别在 ~/lumen 主树工作 —— 有并行会话!)
git -C ~/lumen worktree add --detach /tmp/lumen-dog origin/main
cd /tmp/lumen-dog && GOTOOLCHAIN=local GOFLAGS=-mod=mod go build -o bin/lumen ./cmd/lumen
# 2) 提供 DeepSeek key(committed lumen.toml 用 api_key_env=DEEPSEEK_API_KEY;inline key 在 ~/.config/lumen)
export DEEPSEEK_API_KEY=$(grep -E '^api_key' ~/.config/lumen/lumen.toml | head -1 | sed -E 's/.*"(.*)".*/\1/')
# 3) 在一个 scratch 项目里跑(CWD 必须有 lumen.toml —— 它不从 fresh cwd 读 ~/.config/lumen)
mkdir -p /tmp/dog && cd /tmp/dog && cp ~/lumen/lumen.toml ./lumen.toml
GOTOOLCHAIN=local /tmp/lumen-dog/bin/lumen run "<真实多步任务>"
```

**DoD-级 dogfood**:在 Lumen 自身代码的隔离副本上让它做真实改动(「用 Lumen 改 Lumen」),观察体验毛刺 + 纠错质量。

## 5. 建议的 ultracode 打法(ultracode = 穷尽式多 agent workflow)

1. **Phase 1 — 体验审计 workflow**:对每个 UI/渲染/输入面(TUI 三栏、状态栏、diff、流式、lineedit、首启、markdown、配色、宽字符)扇出 agent,编目每一处像素级瑕疵 → 对抗验证 → 排序。**渲染类必须配真实 `lumen run` 输出捕获来佐证**(纯静态读代码会漏视觉问题)。
2. **Phase 2 — 纠错鲁棒性 workflow**:对每个失败模式 + verify-repair 循环扇出 → 找「错误不清晰 / 不可恢复 / 自纠失败」的点。
3. **Phase 3 — Dogfood gauntlet**:用 Lumen 跑一批真实硬任务(多文件、重构、修失败测试、边界),从真实使用里收集体验摩擦。
4. **Phase 4 — 系统性像素打磨**:逐条修编目的瑕疵,TDD,逐个验证。
- 每阶段:独立 worktree、TDD、`make check` + `-race` 绿、单一关注点 PR、趁网络窗口合并。

## 6. 故意留给 ultracode 的两块硬骨头

1. **lineedit 长行光标 redraw**(`internal/lineedit/editor.go`):`redraw()` 的 Step3(单条 `\x1b[NC` 跨不了换行行)+ Step4(`lastRows`)+ Step1(清屏)三者耦合;终端「幽灵列(phantom column)」语义让它很微妙;**改的是最核心的输入路径,且无真终端验证不了** → 本轮不敢盲改。ultracode 正合适:抽出纯 cursor-math 辅助函数穷尽测试 + 在真 tty 上验证。体验价值高(打长输入时光标跳错行)。
2. **并行工具 `✓` 错位**(`cmd/lumen/terminal.go` 非交互 sink):ToolDispatch 开一行不换行、ToolResult 追加 `✓` 的模式,对顺序工具完美,但并行只读工具(glob/read/grep 批量)时,下一个 dispatch 的 `\n` 会在 `✓` 到达前关掉上一行 → `✓` 跑到单独的行。干净修复与「edit 的 diff 必须显示在工具名下方」「慢 bash 要即时显示已启动」耦合 → 需要重构流式 sink。常见(每次多文件读都出现),值得做对。

## 7. 坑(血泪换来的)

- **worktree 隔离**:**有一个并行会话在 `~/lumen` 主树工作**。永远在自己的 detached worktree(`git worktree add --detach /tmp/lumen-X origin/main`)。**推送前必须重新 `git fetch` 并重新核对** —— 网络抖时本地 `origin/main` 会 stale(本轮踩过:基于过期 main 建分支,文件全是旧的)。
- **`make check` = build/vet/test,不卡 gofmt**。很多文件有**既有 gofmt 漂移**(`if {return}` 单行风格)—— **别 `gofmt -w` 整个文件**(会改一堆无关行、撑大 diff);只保证你的新文件 gofmt 干净。
- **`*_<GOOS>_test.go` 陷阱**:文件名如 `detect_js_test.go` 会被当成 GOOS=js(wasm)构建约束 → 在 darwin 构建里被静默排除、测试根本不跑。避免 `_js/_windows/_linux/_arm…` 后缀。
- **TDD 纪律**:红→绿、真测试。Lumen 的线 = `make check` + `-race` 绿。
- **网络**:GitHub API(graphql + rest)间歇 503。`git push` 较稳;建 PR 用 `gh api --method POST repos/exergyleizhou-ux/lumen/pulls`(REST)带重试;合并用 `gh api --method PUT .../pulls/N/merge -f merge_method=squash`。
- **x/ansi**:`#16` 起是直接依赖。宽度一律用 `ansi.StringWidth/Truncate`;换行用 `Hardwrap(Wordwrap())`(`ansi.Wrap` 对无空格中文插空行)。
- **安全债**:DeepSeek api_key 仍在 git **历史**里(仓库树已用 api_key_env 干净)。轮换是**用户专属动作**。
