# 04 — Storm / Verify-after-edit 设计（可实施）

---

## 1. Storm Breaker

### 语义
同一 **tool 名 + 错误签名**（规范化 message 前 N 字符）连续失败 ≥ `threshold`（默认 3）→  
向模型注入系统提醒，要求换策略；可选硬停止本轮继续同工具。

### 伪接口
```text
struct StormBreaker {
  last_sig: Option<String>,
  count: u32,
  threshold: u32, // 3
}
fn on_tool_error(&mut self, tool: &str, err: &str) -> Option<StormAction>
// StormAction::Nudge(String) | StopBatch
```

### 落点候选（实现时选一，优先侵入小）
1. PostToolUse hook 聚合（若 hooks 可注入模型可见文本）  
2. shell/agent 工具结果汇总处  
3. 独立模块 + 在 tool runtime 包装

### 测试
- 同错误 3 次 → 第 3 次出现 nudge 文本  
- 不同错误 → 计数重置

## 2. RepeatSuccessGuard

### 语义
同一 **tool + args 哈希** 连续成功 ≥ 3 → 提醒「勿无意义重复」。

### 测试
- 相同 read 同路径 3 次 → 警告  
- 路径变化 → 重置

## 3. Verify-after-edit

### 触发
本 turn 内发生 **写文件类工具**（search_replace / write 等）且 session 未关闭。

### 流程
```
收集 changed_files
  → Detect(language / project root)
  → 生成 Steps（build/vet/test 按语言）
  → 顺序执行，超时 timeout（默认 30s/步）
  → 失败：Parse diagnostics → 注入模型（下一轮）repair_cycle++
  → repair_cycle >= max（默认 3）→ 停止自修，把失败交给用户
  → 成功：可选短确认，不刷屏
```

### 语言检测（最小）
| 扩展名 | 步骤 |
|--------|------|
| `.go` | `go build` / `go vet` / `go test`（changed pkg） |
| `.py` | `ruff` 若有；`pytest` 相关 |
| `.ts/.tsx/.js` | `tsc --noEmit` 若有；`jest`/`vitest` 若有 |

工具不存在 → **跳过该步，不记失败**。

### 配置（`config.toml` 示意）
```toml
[verify]
enabled = true
max_repair = 3
timeout_secs = 30
scope = "changed-pkg"  # or workspace
```

### 落点
- **推荐：** agent 在 writer 批处理后统一 Verify（类似 Lumen controller）  
- **备选：** PostToolUse 脚本调用 `packs` 里的 verify helper（先 Go 外置再内嵌）

### 不计成本选项
- 先做 **外置 `lumen-verify` CLI**（从 Go editverify 移植/调用）被 hook 调用 → 验证产品行为  
- 再 **Rust 内嵌** 去进程开销  

### 测试
- fixture：坏 Go 文件 → verify 失败 → 模型收到 diagnostics  
- 修到绿 → 不再 repair  
- max_repair 达上限 → 停止

## 4. Delivery / Goal（与 Storm/Verify 同级可实施规格）

> 规格源：Reasonix `delivery_scope.go` / GOAL 文档 + Grok 已有工具  
> `todo_write`、`update_goal`（见 `xai-grok-tools/.../todo`、`.../update_goal`）  
> **Grok 已有 Goal 分类器与 Ack 通道**；Lumen 层只做 **硬闸与证据**，不重写分类器。

### 4.1 状态机（会话级，host-owned，不进 provider prefix）

```text
enum DeliveryStrictness { Off, Soft, Strict }

struct DeliverySessionState {
  turn_id: u64,
  writer_tools_this_turn: u32,      // search_replace / write 等
  verify_ok_this_turn: bool,        // verify-after-edit 或显式 bash 测试成功
  bash_success_with_test_hint: bool,// 结果匹配 (?i)(PASS|ok|0 failed)
  soft_nudge_count: u32,
  goal_incomplete_override_armed: bool, // 已拦截过一次 completed=true
}

fn on_turn_end(state, todos, strictness) -> DeliveryAction
// DeliveryAction::None | InjectSystemReminder(String) | BlockComplete { detail }
```

### 4.2 Delivery 规则

| 条件 | Soft（默认） | Strict |
|------|--------------|--------|
| 本 turn `writer_tools>0` 且 `!verify_ok` 且 `!bash_success_with_test_hint` | `InjectSystemReminder` 一次（每 turn 最多 1） | 同左，且禁止 `update_goal(completed:true)` |
| todos 存在且有 `pending/in_progress`，模型输出「做完了」类收束 | Soft nudge | 强制继续 |
| `delivery = Off` | 不干预 | — |

**Reminder 文案模板（固定字节，利于 cache）：**
```text
<delivery-reminder>
This turn modified files but no successful verification evidence was recorded
(build/test/verify). Run the project checks or cite a successful command before
claiming completion.
</delivery-reminder>
```

### 4.3 Goal 规则（叠在 Grok `update_goal` 上）

Grok 输入（已存在，勿改 schema 除非必要）：

```text
UpdateGoalInput {
  completed: Option<bool>,      // true = 请求结束 goal
  message: Option<String>,
  blocked_reason: Option<String>,
}
// Ack 已由 SessionActor 返回：Accepted | Rejected | Classifier* | DeferredToTurnEnd ...
```

**Lumen 硬闸（在 `UpdateGoalTool::run` 入口或 SessionActor 处理 `completed:true` 之前）：**

```text
fn gate_goal_complete(todos: &TodoState, delivery: &DeliverySessionState, cfg) -> GoalGate {
  if !cfg.goal_todo_gate { return Allow }
  if !has_incomplete_todos(todos) { return Allow }
  match cfg.goal_incomplete_policy {
    SoftOnce => {
      if !delivery.goal_incomplete_override_armed {
        arm override; return Reject {
          reason: "incomplete_todos",
          detail: list_incomplete_todo_ids(todos), // 模型可见
        }
      }
      Allow // 第二次 completed:true 放行（与 Reasonix 默认一致）
    }
    Strict => Reject { ... } // 永不 override，直到 todos 全 completed
  }
}
```

### 4.4 精确注入点（Grok 路径）

| 步骤 | 文件/模块（相对 `agent/`） | 动作 |
|------|---------------------------|------|
| D-in-1 | `crates/codegen/xai-grok-tools/.../todo/mod.rs` | 维护 `TodoState`（已有）；导出 incomplete 查询 API |
| D-in-2 | `crates/codegen/xai-grok-tools/.../update_goal/mod.rs` | `run()` 在发往 SessionActor **前** 调 `gate_goal_complete`；Reject → 工具错误码 + detail |
| D-in-3 | shell session actor（`xai-grok-shell` 内处理 `UpdateGoalEnvelope` 处） | 二次保险：completed 且 gate 失败则 `UpdateGoalAck::Rejected` |
| D-in-4 | 工具结果汇总 / turn 边界（shell agent 循环） | 维护 `DeliverySessionState`；turn end 调 `on_turn_end` |
| D-in-5 | writer 工具成功路径 | `writer_tools_this_turn += 1` |
| D-in-6 | verify-after-edit 成功 / bash 成功且像测试 | 置 `verify_ok` / `bash_success_with_test_hint` |
| D-in-7 | hooks（可选外置，不优先） | 见 4.5；内置优先于外部 hook |

**禁止**把 Delivery 状态写进 system prompt 前缀（破坏 DeepSeek cache）。只进 **turn tail system-reminder** 或 tool result。

### 4.5 Hook 签名（若走 hooks 外置路径）

Grok 已有事件（`xai-grok-hooks` `event.rs`）：

```text
// stdin JSON → HookEventEnvelope { hook_event_name, session_id, cwd, workspace_root, timestamp, payload }

PreToolUse {
  toolName, toolUseId, toolInput, toolInputTruncated,
  permissionMode?, subagentType?
}

PostToolUse {
  toolName, toolUseId, toolInput, toolResult,
  toolInputTruncated, toolResultTruncated, durationMs?, isBackgrounded, subagentType?
}

PostToolUseFailure { toolName, toolUseId, toolInput, toolInputTruncated, error, subagentType? }

// 阻塞 PreToolUse 的决策（result.rs）
enum HookDecision {
  Allow,
  Deny { reason: String, hook_name: String },
}
```

**Delivery 相关 hook 用法（仅作补充，主路径仍是内置 gate）：**

| hook_event | matcher | 行为 |
|------------|---------|------|
| `PostToolUse` | `search_replace` / `write` / `run_terminal_command` | 更新 evidence 侧车文件 `~/.lumen/sessions/<id>/delivery.json` |
| `PreToolUse` | `update_goal` | 读 todos + delivery.json；若 `completed:true` 且 incomplete → **Deny**（Strict 或 Soft 首次） |
| `Stop`（若可用） | — | Soft：写 reminder 到侧车供下一 turn 注入 |

配置示例（`~/.lumen/hooks.json` 形态，以实现为准）：

```json
{
  "PreToolUse": [
    {
      "matcher": "update_goal",
      "hooks": [{ "type": "command", "command": "lumen-delivery-gate", "timeout": 5 }]
    }
  ],
  "PostToolUse": [
    {
      "matcher": "search_replace|run_terminal_command",
      "hooks": [{ "type": "command", "command": "lumen-delivery-evidence", "timeout": 5 }]
    }
  ]
}
```

`lumen-delivery-gate`：读 stdin envelope → 解析 `toolInput.completed` → exit 2 + reason = Deny（或按 Grok hook 协议写 stdout JSON；**实现时对照 hooks runner 文档定 exit code**）。

### 4.6 配置

```toml
[delivery]
strictness = "soft"          # off | soft | strict
goal_todo_gate = true
goal_incomplete_policy = "soft_once"  # soft_once | strict
max_soft_nudges_per_turn = 1
```

### 4.7 测试（必须）

| ID | 场景 | 期望 |
|----|------|------|
| DG1 | 改文件、无 verify/bash 测试、soft | turn end 出现 delivery-reminder |
| DG2 | 改文件后 verify 绿 | 无 reminder |
| DG3 | todos 未完 + `update_goal(completed:true)` 首次 | Reject incomplete_todos |
| DG4 | 同 DG3 第二次 completed | Soft 放行 / Strict 仍拒 |
| DG5 | delivery=off | 全部放行 |
| DG6 | 状态不进入 system 前缀 | cache shape 审计通过 |

## 5. 与「体验」的平衡

| 机制 | 默认 | 可关 |
|------|------|------|
| 安全 L0–L3 | 开 | 不可关 deny；可关部分 ask 噪音 |
| Storm | 开 | config |
| Verify | 开（有工具链时） | config |
| Delivery 严格 | 中 | config |
| Guardian LLM | **关** | v1.1 |

默认要 **保护生产力**：verify 失败要清晰、可跳过单次（用户命令），避免死循环卡死会话。
