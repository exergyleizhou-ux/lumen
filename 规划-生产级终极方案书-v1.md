## 0. 目标与北极星

### 0.1 一句话目标
**把 Lumen 做成"在终端里可靠写出好代码、用着舒服"的生产级 coding agent——既是我每天的主力工具,也是绿洲 Oasis 上写 C2D 算法的官方作者工具链。**

### 0.2 双目标定义(都要做到生产级)
- **目标 A(自用主力):** 对标 Claude Code 的终端体验。我每天真在用它写代码、改 bug、跑测试,而不是 demo。
- **目标 C(Oasis C2D 工具链):** 用 Lumen 稳定产出符合契约的 C2D 算法(`docker run --network none`,`/data` ro → `/out/output.bin`),并与 Oasis 沙箱/凭证闭环对接、随市场分发。

> **明确不做(本周期):** 面向不信任外部用户的公开开源分发(跨平台安装器、插件市场、多租户安全、社区运营)。留作 v2。这让范围聚焦,跑得快。

### 0.3 北极星指标(可度量,见 §6)
1. **编码正确率** — agent 改完一个任务后,代码能编译 + 相关测试绿的比例(verify-after-edit 的通过率)。
2. **可靠性** — 全量 `go test -race ./...` 长期绿;无崩溃/死锁/flake;编译门常绿。
3. **体验** — 首次安装到跑通的时间;流式/diff/状态/输入的顺滑度;自举(用 Lumen 改 Lumen)的可用度。
4. **C2D 产出率** — 用 Lumen 从零写出并在 Oasis 真实跑通一个 C2D 算法的成功率/耗时。

### 0.4 反模式(立刻停掉)
- 追**行数/包数**(已从 192 砍到 58,继续只减不增——§5)。
- 往 AUDIT/ASSESSMENT 里写夸张结论。
- 不写失败测试就改 bug;不 `-race` 就说"修好了"。

## 4. 分工模型(Claude 难 / DeepSeek-V4 机械)

**给 Claude(难/不可机械化):**
- 触碰 **agent tool-loop / 流式 / 上下文压缩 / checkpoint / verify 编排** 的任何改动。
- **安全**(权限、bash guard、文件边界、密钥)、**架构决策**、**删哪个包**。
- **有歧义的 debug**(竞态根因、LSP 类型、编译断裂根因)。
- 写 **DeepSeek 任务卡** + **review** DeepSeek 产出。

**给 DeepSeek-V4(机械/规格明确/可并行):**
- 按**明确签名/规格**补纯函数 + 表驱动测试 + golden fixtures(如 C-5 的 Detect/Parse/Config,已验证这套协作可行)。
- 语法高亮加语言、模板/脚手架按既定结构补、文档/注释生成。
- 规格清晰的机械重构(重命名、抽函数、按模板补样板)。

**判据一句话:** 答案唯一、可由验收标准机械判定 → DeepSeek;需要判断/取舍/根因 → Claude。
**铁律:** DeepSeek **不碰** agent loop / 安全 / 架构;只动 Claude 点名文件;只加测试或按签名补码;**产出 100% 过 Claude review 才合**。

## 5. 工作流与质量门

- **TDD**:先写失败测试,再实现(bug 先复现再修)。
- **每次改完必跑**:`GOTOOLCHAIN=local GOFLAGS=-mod=mod go build ./... && go vet ./... && go test -race ./<改动包>/`。
- **`-race` 默认**(这是并发 agent)。
- **CI 硬门(P1 首卡)**:`go build ./... && go vet ./... && go test ./...` 失败即红;故意写错能被拦。
- **包数只减不增**:不新增 internal 包,除非 Claude 在卡里明确批准。
- **小步**:一卡一 PR/commit,信息写清"做了什么 + 验收怎么过"。
- **自举(dogfood)**:能用 Lumen 自己做的卡,就用 Lumen 做——这是 A 的活体验收。
- **并发同树纪律**:Claude 与 DeepSeek 同一仓库工作时,各自 `git add` 点名文件,别 `git add -A` 卷进对方在改的代码。
